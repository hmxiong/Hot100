# 系统方向（分布式训练 / 资源调度 / 训推一体）

## 记录格式

- Q：一句话问清楚
  A：拆成“现象/原因/定位/优化/取舍”五段式回答

## 高频 Q&A

- Q：训练显存由哪些部分组成？哪个最容易被忽视？
   A：常驻：参数、梯度、优化器状态（Adam m/v）；随 batch/seq 增长：激活；另外还有通信 buffer、算子 workspace、dataloader staging。最容易忽视的是通信/分桶 buffer、以及 checkpoint/activation 的临时峰值。

- Q：为什么 Adam 的显存压力特别大？如何缓解？
   A：Adam 需要 m/v 两份状态，常 FP32，等价于每个参数额外 ~8B；再加上 FP32 master weights 时更大。缓解：ZeRO-1 分片 optimizer states；用更省状态的优化器（如 Adafactor 视情况）；offload 到 CPU/NVMe（trade-off 是带宽与抖动）；或者用更小模型并行策略降低单卡驻留。
- Q：Muon 和 AdamW 的 pretrain/post-train 为什么通常不建议混用？
   A：本质是“优化器定义的更新几何 + 状态”不兼容，直接切换会带来一次很大的分布外训练扰动，经验上容易损失收敛性或效果；Muon 相关实证也指出 pretrain 与 SFT 的优化器互换通常表现不好。
  
  - 更新规则差异导致的突变：
     1. AdamW 是自适应一阶法：用一阶/二阶矩（m,v）对每个参数做自适应缩放，等价于改变了每个维度的有效学习率；
     2. Muon 更接近 momentum SGD，但会对 2D 参数的动量更新做近似正交化（Newton-Schulz），更新方向的谱性质与 AdamW 完全不同。
  
  - 优化器状态不兼容：
     1. 从 AdamW → Muon：AdamW 的 v（方差估计）等状态无法“平滑迁移”到 Muon 的状态表述；如果硬切通常要丢状态，相当于突然换了优化器 + 重置动量，容易 loss spike；
     2. 从 Muon → AdamW：同样缺少可复用的二阶统计，AdamW 刚接上时自适应缩放不稳定，需要重新 warmup/重调参。
  
  - 超参是在“阶段 + 优化器”耦合调出来的：
     1. pretrain（海量 token、噪声大）与 post-train/SFT（数据更窄、更偏好对齐）最优 lr/wd/betas 往往不同；
     2. Muon 常与“辅助 AdamW”混用来优化 embedding/输出层/1D 参数，本身就引入分组学习率/策略；换成纯 AdamW（或反过来）需要重做分组与 lr 匹配。
  
  - 工程建议（必须切换时）：
     1. 切换点做短跑校准：先用小步数验证 loss/梯度范数是否稳定；
     2. 显式重置或渐进衰减旧 optimizer states，并重新 warmup 学习率；
     3. 重新调 weight decay 与分组策略（尤其 embedding/head/Norm 等参数组）。
- Q：ZeRO-1/2/3（或 FSDP）各自解决什么？代价是什么？
   A：
  
  - ZeRO-1：分片 optimizer states，省常驻内存，通信增加不大；
  - ZeRO-2：再分片 grads，进一步省；
  - ZeRO-3：再分片 params，最省，但需要频繁 all-gather/reshard，通信、实现复杂度与 tail latency 风险更高。
     选择原则：先用 1/2 满足需求，逼不得已再上 3，并确保网络与实现成熟。
   关于ZeRO-1/2/3对于实际显存的节省估算：
   fp16权重1.5*2=3G
   fp16梯度1.5*2=3G
   fp32优化器权重 1.5*4=6G
   fp32优化器一阶动量1.5*4=6G
   fp32优化器二阶动量1.5*4=6G

- Q：DP/TP/PP 怎么选？如果模型放不下你会先动哪一个？
   A：放不下优先 TP 或 PP：
  
  - 层太大（如 attention/MLP 单层权重/激活峰值）：优先 TP；
  - 层数太深导致总参数/激活过大：优先 PP；
     DP 是“放得下之后扩吞吐”的首选。最终组合看通信拓扑与实现：TP 通信频繁，对 NVLink/IB 更敏感；PP 要处理 bubble 与调度复杂度。
- Q：为什么 PP 会有 bubble？如何减少？
   A：流水线需要填充/排空，micro-batch 太少或 stage 不均衡会产生 bubble。减少方式：增加 micro-batch 数、做更均匀的 stage 切分、用 interleaving（交错流水）提升利用率。trade-off 是更多 micro-batch 会增加 activation/调度开销。
- Q：activation checkpointing 的正确使用姿势？
   A：在显存紧张且算力有余时开启；优先对占用最大的模块做 checkpoint（如 attention/MLP block）；配合 flash-attn/sequence parallel 等进一步降激活。注意不要把计算打爆导致吞吐显著下降，最好用 profiler 量化 gain（能换来更大 batch/seq 才值得）。
- Q：混合精度训练常见数值问题和排查手段？
   A：溢出/下溢、loss 变 NaN、梯度异常。手段：动态 loss scaling、关键归约用 FP32、softmax 做 max-subtract、检查 layernorm/softmax 的实现、监控梯度范数与激活统计；定位用二分法：逐层 hook 看何处出现 NaN/Inf。
- Q：分布式训练“慢”的典型原因有哪些？
   A：通信占比过高（AllReduce/AllGather）、bucket 太小频繁启动、拓扑不匹配（跨机流量大）、数据管道慢导致 GPU idle、以及 kernel 效率低。定位：step time 拆 compute/comm/data；看 NCCL 时间、链路带宽、GPU idle；再优化桶大小、重叠通信与计算、拓扑感知 placement、数据预取与缓存。
- Q：如何做通信与计算重叠（overlap）？
   A：在反向传播过程中，把梯度分桶，某些 bucket 梯度 ready 就立刻启动 AllReduce，与后续层的反向计算并行；前提是框架支持异步通信、合理的 bucket size、以及 stream/依赖管理正确。trade-off：桶太大 overlap 变差，太小 launch 开销大。
- Q：NCCL 中的通信原语有哪些？用 AllReduce 做一次参数更新需要几次通信？
   A：常见 collective 与 P2P 如下；“几次通信”取决于你按 API 还是按算法步骤计数：
  
  - 原语（collective）：
     1. AllReduce（所有参与者聚合并广播结果）
     2. Reduce（聚合到 root）
     3. Broadcast（root 广播到所有）
     4. AllGather（每个参与者收集他人数据的串接）
     5. ReduceScatter（先聚合再分片发回）
     6. AllToAll（每对互相发送等量数据）
     7. Barrier（同步）
     8. （较新版本）点对点 Send/Recv（配合 group 使用）
  
  - AllReduce 计数口径：
     1. API 口径：一次参数更新对应每个梯度桶触发 1 个 AllReduce；
     2. 算法口径：NCCL 的环形 AllReduce 通常拆成 ReduceScatter + AllGather 两个阶段；如果细到环步，是 2×(WorldSize-1) 个环步；
     3. 变体：很多实现直接用 ReduceScatter 平均梯度 + 本地优化更新，再在需要时 AllGather 权重（或后续再同步）。ZeRO-2 用 RS 替代 AR；ZeRO-3/FSDP 则在前向前做参数 AllGather、反向后做梯度 ReduceScatter，通信次数相对更多但驻留更低。

- Q：megatron-lm 中通信优化怎么做？
   A：先按并行维度拆通信，再讲“减少通信量 + 减少通信次数 + 隐藏通信时间 + 拓扑/实现细节”四类手段：
  
  - 通信长什么样（先把面试官对齐）：
     1. DP：梯度 AllReduce/ReduceScatter（bucket 化）；
     2. TP：层内高频 collective（典型是 AllReduce / AllGather / ReduceScatter，取决于张量切分方式与算子实现）；
     3. PP：stage 之间激活/梯度的 Send/Recv（pipeline schedule 决定频次与 bubble）。
  
  - 减少通信量（bytes）：
     1. 用更省通信的并行组合：能提高 DP 就少加 TP（TP 的通信频率最高）；PP 用合理的 stage 切分降低跨 stage 激活大小；
     2. Sequence parallel：把某些按序列维度的中间激活/归约切分到 TP rank，减少 TP 的 activation 通信压力（注意对 layernorm/dropout 等实现约束）；
     3. 通信压缩/低精度：梯度通信用 BF16/FP16（或更激进的压缩在特定场景），但要关注收敛与数值稳定。
  
  - 减少通信次数（launches）：
     1. 梯度分桶（bucketization）：把小 tensor 聚成大桶再做 collective，减少 NCCL 启动开销；
     2. 合并/融合通信：同一通信组内尽量合并多个小 all-reduce；TP 侧尽量用 ReduceScatter+AllGather 取代多次小 AllReduce（视算子路径而定）；
     3. CUDA Graph：把每 step 的通信+计算捕获进 graph，降低 launch overhead（对 shape 稳定更友好）。
  
  - 隐藏通信时间（overlap）：
     1. DP overlap-grad-reduce：反向时梯度 ready 就立刻启动桶级 AllReduce/ReduceScatter，与后续层的 backward 计算并行；
     2. TP overlap：把层内 RS/AG 放到独立 stream，并用 event 管依赖；结合 fused kernel/epilogue，减少必须同步的边界；
     3. PP：用 1F1B / interleaving 等 schedule 降 bubble，并让 send/recv 与计算尽量并行（依赖正确是关键）。
  
  - 拓扑与实现（经常决定成败）：
     1. 通信组映射到硬件拓扑：TP 尽量落在 NVLink/NVSwitch 域内；跨机通信尽量留给 DP（如果网络足够强）；
     2. NCCL 调参：合理的 channel/algorithm/协议选择、启用/禁用特定路径（IB/NVLink/SHARP 视环境），避免小消息频繁同步；
     3. Profiling 方法：把 step time 拆 compute/comm；看 NCCL time、链路带宽、bucket size 分布、以及 overlap 后 GPU idle 是否下降；目标是提高 MFU 而不是“通信时间单项变小”。

- Q：检查点（checkpoint）如何设计，才能支持抢占恢复（preemption）？
   A：保存模型权重、优化器状态、lr scheduler 状态、随机种子、数据迭代进度（epoch/step、dataloader sampler 状态）。写入要异步/分片化减少停顿；恢复要校验版本与一致性。若做训推混部抢占，必须保证“可恢复 + 可重复”。

- Q：训推一体 / 潮汐调度的核心策略怎么讲？
   A：推理优先保证 SLA（低延迟、稳定）；训练允许抢占与回填。调度器做多队列与优先级：白天为推理预留容量，训练使用可抢占队列；夜间释放给训练。关键是隔离（cgroup/容器/GPU 配额）、快速弹性（权重分发/热启动）、以及抢占恢复链路（checkpoint 频率与成本平衡）。

- Q：多租户集群如何避免“邻居噪声”（noisy neighbor）？
   A：资源隔离：CPU/内存/IO cgroup，GPU 级别配额与并发限制；网络与存储 QoS；调度策略上按租户配额/权重公平；观测上给每租户提供可归因指标（队列等待、GPU 时间、IO 带宽），发现噪声能快速定位与限流。

- Q：如何衡量训练系统的效率？MFU 是什么？
   A：常用 MFU（Model FLOPs Utilization）：实际训练吞吐对应的有效 FLOPs / 理论峰值 FLOPs。MFU 低通常意味着通信/IO/低效 kernel 或 bubble 多。面试回答要能说：如何估算模型 FLOPs、如何从 profiler 得到 step time、以及优化路径（通信重叠、flash-attn、并行策略、数据管道）。

- Q：给定训练所需的 Tokens，怎么估计模型训练所需的完整时间？
   A：两条常用口径，优先用“实测吞吐”更准：
  
  - 吞吐法（最常用）：总训练时长 ≈ T_total / (tokens/s)_eff。
     (tokens/s)_eff ≈ (global_batch_tokens / step_time) × utilization，其中 global_batch_tokens = global_batch_size × seq_len × valid_token_ratio；utilization 用来吸收评测、checkpoint、容灾重启、排队抖动等非纯训练时间（经验上用 0.7–0.95 的区间做上下界）。
     实操：用小规模/短跑拿到稳定的 step_time 与 valid_token_ratio（padding/packing 后的有效 token），然后按目标集群规模与并行策略复现同等负载，再线性外推到目标 tokens。
  
  - FLOPs 法（规划/对账用）：总训练时长 ≈ FLOPs_total / (N_gpu × FLOPs_peak_gpu × MFU)。
     对稠密 Transformer（decoder-only）常用近似：FLOPs_total ≈ 6 × N_params × T_total；MFU 直接吸收通信、bubble、低效 kernel、重计算（activation checkpointing）等损耗。
     注意：如果是 MoE、长序列导致 attention 占比变化、启用重计算、或存在频繁 eval/save，会显著偏离该近似，需要用实际 profiler 或线上吞吐校准。

- Q：如果要求你用 PyTorch 从 0 实现一个简化版 FSDP/参数分片，你会怎么做？
   A：思路：把参数按 rank 分片存储；前向需要用到参数时 all-gather 到本 rank 的 full param（或用 flat buffer），计算后释放；反向得到 grad 后 reduce-scatter 分片存回；优化器只维护本 rank 的 shard 状态；并确保 autograd hook 与 stream 同步正确。trade-off：all-gather/reshard 频繁，需 bucket 化与 overlap 才能跑得动。

- Q：在 AI 训练场景下，如何设计和实现一个高性能的参数服务器（Parameter Server）？需要考虑哪些关键点？
  A：参数服务器本质是在做“分布式参数/梯度的存储、聚合与一致性协议”。高性能的核心是把系统做成：参数分片（shard）+ 高吞吐网络路径（RDMA/高并发 RPC）+ 有效的聚合与缓存策略，并针对训练负载（dense vs sparse、同步 vs 异步）做取舍。
  
  - 现象（问题长什么样）：
    1. worker 端训练 step time 被 push/pull 参数/梯度拖慢，GPU 出现周期性 idle；
    2. 参数节点 CPU/网卡打满但吞吐上不去，tail latency 高，收敛不稳定（异步场景更明显）；
    3. 大量小 tensor 请求导致 RPC/序列化开销主导，或热点参数（如 embedding 热 id）导致单 shard 过载。
  
  - 原因（瓶颈从哪来）：
    1. 通信模式：PS 是多对一/多对多的聚合，容易形成网络/CPU 聚合热点；
    2. 张量颗粒度：小消息多、频繁同步导致协议与调度开销大；
    3. 一致性与更新策略：同步 PS 受 straggler 影响，异步 PS 受 staleness 影响；
    4. 参数形态：稠密参数适合 AllReduce，稀疏/embedding 更适合 PS（只传非零/命中 id）。
  
  - 定位（怎么证明瓶颈在 PS）：
    1. 拆分 step time：compute / comm / input，观察 comm 是否与 PS 请求强相关；
    2. PS 侧指标：每 shard QPS、吞吐（bytes/s）、P95/P99 延迟、CPU 使用率、网卡利用率、队列长度、热点 key 分布；
    3. worker 侧指标：pull 等待时间、push 等待时间、batch size 与消息大小分布、重试/超时；
    4. 网络侧：RDMA/IB 利用率、拥塞、跨机架流量占比与包重传。
  
  - 优化（工程上怎么做快）：
    1. 架构选型：参数按 key-range 或 hash 分片到多台 PS；支持 replica/primary-backup 或 chain replication；必要时按层/参数组做层级 PS（机内聚合→机架→跨机架）；
    2. 通信协议：避免 Python/解释器路径；用 C++/Rust 实现核心 RPC；支持零拷贝、批量化（coalescing）、长连接、多路复用；优先 RDMA/GPUDirect（条件允许）减少 CPU 开销；
    3. 聚合与压缩：server-side aggregation（在 PS 聚合多个 worker 的梯度）；梯度分桶；稀疏梯度用 top-k/CSR；可选低精度通信（BF16/FP16）或误差反馈压缩；
    4. 一致性策略：同步训练用 barrier/SSP（stale synchronous parallel）降低 straggler；异步训练用 bounded staleness + 学习率/动量修正；
    5. 热点与负载均衡：对热点 embedding 做 key 拆分/复制（hot shard replication）或一致性 hashing 动态迁移；对请求做限流与优先级；
    6. 内存与存储：参数用 flat buffer/内存池；热参数缓存；冷参数分层到 CPU/NVMe；checkpoint 增量化与异步化；
    7. 容错与弹性：PS 节点故障恢复（日志/快照/重放）、worker 重试与幂等更新、弹性扩缩容（reshard）与训练不中断或可恢复。
  
  - 取舍（面试官最爱追问的 trade-off）：
    1. PS vs AllReduce：稠密大模型通常 AllReduce 更简单且更高效；PS 更适合稀疏更新/embedding/异构资源或需要参数分层存储的场景；
    2. 同步 vs 异步：同步稳定但慢点在 straggler；异步快但要处理 staleness、可能影响收敛与最终精度；
    3. 复制 vs 分片：复制提升读吞吐与容错但增加写一致性与内存；纯分片内存省但热点与故障更敏感；
    4. 压缩 vs 精度：通信压缩省带宽但可能引入偏差，需要误差反馈/校准与收敛验证。
