# 算子方向（CUDA / 推理加速 / 推理引擎）

## 记录格式

- Q：一句话问清楚
  A：先给结论，再给关键原因与工程取舍；必要时补 profiler 指标/roofline 口径

## 高频 Q&A

- Q：prefill vs decode 的本质差别？为什么 decode 难做满 GPU？
   A：prefill 一次处理整段 prompt，矩阵计算规模大，偏 compute；decode 每步只新增 1 token，单步算子更碎、batch 波动大，容易被 kernel launch overhead、KV cache 访存带宽、以及调度策略限制，导致 SM 利用率不稳。工程上靠 continuous batching、KV 管理、算子融合/减少 launch 次数，把 decode 做“更像大算子”。
- Q：KV cache 为什么会成为主要瓶颈？
   A：decode 每一步 attention 都要读取历史 token 的 K/V，序列越长 KV 越大、访问越频繁，整体更像 memory-bound；同时 KV 的分配/碎片、以及不同请求长度导致的浪费会进一步拉高显存压力。核心优化方向：块/页管理减少碎片、提高复用；在保证精度下做压缩/量化；调度上减少无效搬运与重排。
- Q：DeepSeek 的 DSA、NSA 和 Kimi 的 MoBA 有什么区别？
   A：它们都是“稀疏注意力（sparse attention）”路线，但稀疏的组织方式不同：NSA 用“分层 token 选择（粗压缩+细选择）”，MoBA 用“按 block 切分+MoE 式路由选择 block”，DSA 是 DeepSeek 在 V3.2 中落地的稀疏注意力机制，目标是在长上下文下显著降复杂度同时尽量不掉效果。
  
  - NSA（Native Sparse Attention，DeepSeek）：
     1. 关键词：natively trainable、hardware-aligned、dynamic hierarchical sparse strategy；
     2. 稀疏方式：结合粗粒度 token 压缩与细粒度 token 选择，兼顾全局感知与局部精细度；
     3. 目标：训练/推理全链路都加速（forward/backward/decoding），并强调算术强度与硬件实现匹配。
     参考：[NSA: Hardware-Aligned and Natively Trainable Sparse Attention](https://arxiv.org/abs/2502.11089)
  
  - MoBA（Mixture of Block Attention，Moonshot/Kimi）：
     1. 关键词：MoE 思想迁移到 attention、“less structure”（尽量少手工结构偏置）；
     2. 稀疏方式：把序列分成 blocks，用路由机制选择需要参与注意力计算的 blocks；
     3. 亮点：可以在 full attention 与 sparse attention 之间平滑切换，降低“稀疏结构选错导致性能崩”的风险。
     参考：[MoBA: Mixture of Block Attention for Long-Context LLMs](https://arxiv.org/abs/2502.13189)
  
  - DSA（DeepSeek Sparse Attention，DeepSeek-V3.2）：
     1. 关键词：面向长上下文的高效注意力、显著降低计算复杂度、尽量保持模型表现；
     2. 定位：更像是“在大模型产品/大规模训练集群上可落地”的稀疏注意力方案（从公开信息看用于 V3.2-Exp 并带来显著成本下降）；
     3. 对比点：相对 NSA 更偏“工程化落地与成本/吞吐收益”，相对 MoBA 不强调 block 路由的 MoE 化表述，而是强调 fine-grained sparse attention 的整体效果与可部署性。
     参考：[DeepSeek-V3.2: Pushing the Frontier of Open LLMs（DSA）](https://arxiv.org/abs/2512.02556)，[DeepSeek API News: V3.2-Exp & DSA](https://api-docs.deepseek.com/news/news250929)
- Q：continuous batching 解决什么？核心实现点是什么？
   A：解决在线请求到达随机、固定 batch 导致等待和 GPU 空转的问题。实现关键：调度器把新请求插入正在执行的 batch；KV cache 需要支持不同序列长度的拼接与增长；必要时支持抢占/回退（例如长请求影响 SLA 时把它降权或暂停）。指标同时盯吞吐（tokens/s）和 TTFT/P99。
- Q：PD 分离（prefill/decode 分离）为什么有效？什么时候不划算？
   A：两阶段资源形态不同：prefill compute-heavy、decode memory/调度-heavy，拆开后可分别用不同 batch/并行策略/实例规格优化，并避免长 prompt 把 decode 队列堵住。不划算的情况：请求很短、并发低、拆分引入跨实例通信/转发开销反而更大。
- Q：多机 PD 分离会有 KV cache transfer 开销，为什么还要做 PD 分离？
   A：因为 PD 分离解决的是“系统级瓶颈形态不一致 + tail latency 不稳定”的大问题，很多线上场景里它带来的吞吐与 P99 收益能覆盖 KV 迁移成本，甚至把原本无法达标的 SLA 变为可达标。
  
  - KV transfer 不是每步都发生：
     1. 通常只发生在 prefill 完成、请求从 P 侧切到 D 侧时；之后 decode 期间请求粘在同一 D 实例，不会每 token 迁移；
     2. 实现上还会尽量做会话粘性（sticky routing），避免重复迁移。
  
  - 资源形态拆开后更容易“各自跑满”：
     1. Prefill 更像大 GEMM/attention，偏 compute；Decode 更像 KV 读写 + 小算子链，偏 memory/调度；
     2. 混在同一批里会互相拖累：长 prompt 的 prefill 会阻塞 decode 队列，直接抬高 TTFT/P99；
     3. 分离后 P 实例用更大 batch/更激进融合提升 tokens/s，D 实例用更强的 KV 管理与小 batch 策略稳住 tail。
  
  - 更好的调度与扩缩容自由度：
     1. P/D 两侧可独立扩缩容，适配流量结构（prompt 长短、并发变化）；混部时很难同时把 TTFT 与吞吐都调到最优；
     2. 队列隔离：把“prefill 抖动”与“decode SLA”隔离开，降低尾延迟放大效应。
  
  - KV transfer 成本可被工程手段显著压低：
     1. 只传增量元数据/按块传输（paged KV），避免大块连续拷贝与碎片；
     2. NVLink/IB/RDMA 做 GPU-GPU 直传（GPUDirect），并用异步 copy 与 compute overlap；
     3. 对长 prompt：prefill 侧做 chunked prefill，把 KV 增长与传输分段化，降低瞬时带宽尖峰与抖动。
  
  - 什么时候“不值得”做多机 PD 分离：
     1. 请求很短或并发很低（prefill 很快），KV transfer 的固定开销占比过大；
     2. 网络带宽/延迟差（跨机 KV 迁移成为瓶颈），或无法做到可靠的 GPU-GPU 直传；
     3. 已经能通过同机两流/同机 PD 分离/更强 batching 把 SLA 跑稳，此时跨机分离复杂度收益比下降。
- Q：如何看待跨 SM 的 PD 分离和 AF 分离？
   A：把它们当作“同一张 GPU 上的空间/并发隔离”手段：目标不是改变算子本身，而是减少 prefill/attention/FFN 之间的资源互相抢占，让吞吐与尾延迟更可控；但前提是你确实有可重叠的空洞，否则很容易变成负优化。
  
  - 跨 SM 的 PD 分离（同卡把一部分 SM 留给 prefill，一部分留给 decode）：
     1. 价值：避免 prefill 的大算子在高峰期把 decode 挤到排队，从而稳 TTFT/P99；同时 decode 可以保持更稳定的 batch/时序，减少抖动；
     2. 代价：把一张卡“硬切资源”会牺牲峰值吞吐（因为单边拿不到全卡资源），并引入调度与隔离开销；
     3. 适用：在线服务、混部场景、请求长度分布很长尾、且 decode SLA 很硬时；
     4. 不适用：单一阶段就能把 GPU 跑满（compute 或带宽已饱和）、并发不够、或 prefill 很少/很短时（切分导致闲置）。
  
  - AF 分离（把 Attention 阶段与 FFN/MLP 阶段做并发隔离/流水）：
     1. 直觉：attention 往往更 memory-bound（KV 访问/softmax/读写），FFN/MLP 更偏 compute/TensorCore；把两类工作并发跑，理论上能让“带宽与算力”更同时被利用；
     2. 真正能跑起来的前提：必须有跨请求/跨 microbatch 的可并行性（例如不同请求处在不同层/不同阶段），并且实现能把两类 kernel 在资源上隔离开，不然还是会在 HBM/L2/SM 上互相干扰；
     3. 常见负优化来源：两边最终都在抢 HBM（尤其 attention 已经用了 FlashAttention/TensorCore 后并不“纯内存”）、或者为了并发把 batch 拆碎导致 GEMM 变小、融合机会变差，吞吐反而下降，P99 还更抖。
  
  - 面试加分的评价口径（如何判断值不值得）：
     1. 看瓶颈：profile 先确认是“阶段互相干扰导致 tail 上升”还是“单阶段资源已满”；
     2. 看资源：同时观察 SM busy、HBM 吞吐、L2 命中与 kernel 分布，判断并发后是互补还是争用；
     3. 看指标：吞吐（tokens/s）+ TTFT/P99 一起看，很多隔离策略会提升 SLA 但牺牲峰值吞吐，需要业务权衡。
- Q：如何设计推理调度以同时优化吞吐和时延？
   A：先定义 SLA（TTFT、P99、tokens/s），再做分层：入口按租户/优先级/长度分队列；prefill 与 decode 分别设置合并窗口（max wait）；对长请求做分级（比如限制最大并发、或单独队列）；必要时按“最短剩余时间/加权轮询”实现公平与稳定。用压测看 tail latency，调参遵循“先保 SLA，再榨吞吐”。
- Q：写一个高性能 GEMM kernel，你会怎么从 0 讲清楚设计？
   A：从 Roofline 讲起：目标是提高算术强度与数据复用。做法：
  
  1. 分块：按 (M,N,K) tile，block 负责一个 C tile；
  2. 数据搬运：A/B tile 用 vectorized load 搬到 shared memory，保证 coalescing，处理 bank conflict；
  3. 计算：warp 级别做子 tile，寄存器累加，利用 Tensor Cores（若允许）；
  4. pipeline：用双缓冲/多 stage 隐藏内存延迟；
  5. 约束：寄存器/共享内存占用影响 occupancy，要在吞吐与并发间平衡。
      验证：用 profiler 看 SM 利用率、DRAM throughput、以及 achieved FLOPs 对比峰值。
- Q：将 Ampere 架构的算子适配到 Hopper（H100）上，你会升级哪些点？
   A：按“算子实现路径 + 数据类型 + 搬运方式 + 调度模型”四层做升级：先把 Tensor Core 路径、异步搬运与 pipeline、以及 block/warp 映射重新 autotune 一遍，再做 Hopper 特有能力（WGMMA/TMA/cluster）的深度重写。
  
  - 计算侧（Tensor Cores 路径升级）：
     1. 引入 FP8（E4M3/E5M2）训练/推理路径（能接受量化误差的算子优先，如 GEMM/FFN/部分 attention 投影），提升峰值 FLOPs；
     2. 用 Hopper 的 WGMMA（warp-group MMA）重做主循环：更大 tile、更深 pipeline，减少指令与同步开销；
     3. 重新选择 MMA shape、accumulator 精度与 epilogue 融合策略（bias/gelu/swiglu/残差等），避免额外写回。
  
  - 搬运侧（global→shared 的方式升级）：
     1. 从 Ampere 的 cp.async 思路扩展到 Hopper 的 TMA（Tensor Memory Accelerator）：用 bulk copy、multicast、以及更强的异步 pipeline 降低搬运/指令开销；
     2. 重新设计 shared memory layout（含 padding）以匹配更大 tile 与更高带宽，继续压 bank conflict；
     3. 关注 L2/SMEM 复用：长序列/重复访问的场景优先做数据驻留与复用，减少 HBM 往返。
  
  - 调度侧（并发/隔离/集群能力）：
     1. 重新 autotune block 大小、stage 数、寄存器与 shared memory 占用，利用 Hopper 的资源上限把 occupancy 与 ILP 拉满；
     2. 需要跨 CTA 协作/更稳定的数据复用时，引入 thread-block cluster（CTA cluster）与 cluster 级同步/共享能力；
     3. 对 kernel launch overhead 敏感的链路（decode、小算子链），更积极使用 CUDA Graph / fusion / persistent kernel。
  
  - 工程侧（实现与验证）：
     1. 编译与多架构分发：明确区分 sm80/sm86 与 sm90/sm90a 的 code path（不同内联 PTX/内建指令集）；
     2. 优先复用成熟库：GEMM 用 CUTLASS 3 / cuBLASLt 的 Hopper 优化 path，attention 复用 FlashAttention Hopper kernel；
     3. 验证以 Nsight Compute 为准：看 Tensor Core 利用率、pipe 结构、HBM/L2 吞吐、stall 原因与 occupancy，避免“峰值更高但实际更慢”的回归。
- Q：如何判断 kernel 是 memory-bound 还是 compute-bound？
   A：看算术强度 + profiler：如果 DRAM 带宽接近峰值且 SM pipeline 不满，多为 memory-bound；如果带宽不高但 FLOPs/SM 活跃高，趋向 compute-bound。用 Roofline 对齐：实际点落在带宽斜线附近就是 memory-bound。
- Q：reduce/softmax/variance 这类算子你会如何兼顾速度与数值稳定？
   A：稳定性：softmax 先减 max，累加用 FP32；方差用 Welford；混合精度配 loss scaling。性能：warp-level primitives（shuffle）减少 shared mem 与同步；分层归约避免全局原子争用；必要时融合（如 bias+mask+softmax）。trade-off 是融合提高性能但可维护性差、对不同 shape 兼容成本高。
- Q：CUDA 中 warp divergence 为什么致命？怎么规避？
   A：warp 是 SIMT 调度单位，同 warp 分支会串行执行不同路径，吞吐下降。规避：数据重排让同 warp 数据同质；用 predication/表驱动替代复杂分支；将分支外提到 block 级别或分 kernel。
- Q：什么是 warp？为什么它对性能这么关键？
   A：warp 是 NVIDIA GPU 的基本执行单元：一组线程（通常 32 个）以 SIMT 方式“锁步”执行同一条指令流；硬件调度与发射以 warp 为粒度。关键点：同 warp 的线程分支不一致会导致串行化；访存模式不一致会导致更多内存事务与吞吐下降。
  
  - warp、lane、warpSize：
     1. warpSize=32（几乎所有 NVIDIA 架构一致），threadIdx.x % 32 是 lane_id；
     2. 一个 SM 同时驻留多个 warps，靠快速切换 warps 隐藏访存延迟（occupancy 影响可隐藏的延迟上限）。
  
  - warp divergence（分支发散）：
     1. 同 warp 的线程走不同 if/else 路径时，硬件会把两条路径都执行一遍，只是用谓词屏蔽不该写回的线程；
     2. 结果是有效吞吐下降（最坏接近按分支路径数成比例下降）。
  
  - warp 相关的常见优化抓手：
     1. 用 warp shuffle（__shfl_*）做归约/广播，减少 shared memory 与 __syncthreads；
     2. 把不规则控制流前移到 block 级或重排数据，让同 warp 数据“同质”；
     3. 合并访存：让同 warp 线程访问连续地址（coalescing），减少内存事务数。
- Q：CUDA 的 global memory 和 shared memory 访存分别需要注意什么？
   A：global memory 核心是“合并访问 + 减少搬运 + 隐藏延迟”，shared memory 核心是“避免 bank conflict + 正确同步 + 控制资源占用”。
  
  - Global memory（HBM/显存）：
     1. 合并访问（coalescing）：同一 warp 的线程尽量访问连续地址、避免大 stride/散乱 gather；让访存落在尽可能少的内存事务上；
     2. 对齐与向量化：保证地址对齐（如 16B/32B），优先用 vectorized load/store（float4/half2 等）提升有效带宽；
     3. 访问模式：尽量把“按列/按行”重排成顺序读写；必要时通过 transpose/tiling 把不连续访问变成连续；
     4. 读缓存与复用：能复用就用块内共享/寄存器缓存；只读数据可走只读路径（减少污染），并避免反复从 HBM 取同一数据；
     5. 带宽与延迟：memory-bound 场景要降低 bytes/FLOP（提高算术强度），并通过更多并发（合适 block 数/occupancy）隐藏延迟；
     6. 代价点：不合并访存、频繁小 memcpy、以及过多的中间张量写回都会迅速把性能打穿。
  
  - Shared memory（片上 SRAM）：
     1. Bank conflict：线程访问落在同一 bank 会串行化；通过布局调整、padding、或让每线程访问不同 bank 来消除；
     2. 访问粒度：shared memory 也要注意对齐与连续访问，避免“结构体/奇怪 stride”引起隐式冲突；
     3. 同步正确性：写入后被其他线程读取需要 __syncthreads（或更细粒度的 warp 同步）保证可见性，避免 data race；
     4. 资源占用：shared memory 用太多会压 occupancy；tile 尺寸/双缓冲要在复用收益与并发之间平衡；
     5. 替代方案：能用寄存器/warp shuffle 解决的归约/广播，尽量不用 shared memory，减少同步与冲突；
     6. Pipeline：对搬运型算子可用多 stage/双缓冲把 global→shared 的加载与计算重叠（目标是让 shared 成为“缓存”而不是“瓶颈”）。
- Q：PyTorch 里写自定义 CUDA 扩展，你会怎么落地并保证工程质量？
   A：用 C++/CUDA Extension：接口层做 shape/dtype/device 校验；实现层区分 forward/backward，必要时写 autograd Function；性能上提供多 kernel path（不同 dtype/shape）；用单测覆盖正确性（含随机输入、边界），并用基准测试对比 torch baseline；避免在 kernel 里做 malloc/free；确保 stream 语义正确（使用当前 CUDA stream）。
- Q：推理服务如何做可观测性，定位“变慢了”的原因？
   A：分层打点：请求队列等待、prefill 时间、decode 时间、batch size、KV cache 命中/碎片、GPU 利用率、kernel time topN、显存占用、NCCL/PCIe/NVLink 带宽（如有跨卡）。定位遵循：先看队列与 SLA，再看 GPU idle，再深入到 kernel/内存/调度。
- Q：如何降低推理单 token 成本（cost/token）？
   A：三类手段：
  
  1. 算子层：flash-attn、融合、减少 launch、量化/低精度；
  2. 调度层：continuous batching、PD 分离、长度分桶；
  3. 系统层：模型并行与实例编排、热点权重/缓存、A/B 与灰度降低回滚成本。
      评估用 cost/token、QPS@SLA、P99、GPU 利用率与显存水位。
- Q：讲述 INT8、W4A16 的具体流程和应用场景
   A：核心区别：INT8 通常是 W8A8（权重/激活都 8bit），追求更高吞吐但量化链路更复杂；W4A16 是 weight-only（权重 4bit、激活 FP16/BF16），更稳、更常用于 LLM 推理降显存/带宽。
  
  - INT8（典型 W8A8）流程：
     1. 选量化方案：对称/非对称（是否有 zero-point）、per-tensor vs per-channel（权重常 per-output-channel，激活常 per-tensor 或 per-token）。
     2. 离线量化权重：w_int8 = round(w_fp / s_w)（或带 zp），并保存 scale（以及 zp）。
     3. 标定/训练：PTQ 通过 calibration 统计激活范围/分位数得到 s_a；若精度不够可做 QAT（训练时插入 fake-quant）。
     4. 在线量化激活：a_int8 = round(a_fp / s_a)（静态用标定 s_a；动态可按 token/batch 估计范围得到 s_a）。
     5. INT8 GEMM/Conv：int8×int8 → int32 accumulate（Tensor Cores/DP4A），epilogue 融合 bias/激活函数/残差。
     6. Requant/Dequant：输出要么继续保持 int8（跨层 int8 链路），要么 dequant 回 FP16/BF16（更通用，便于与 layernorm/softmax 等 FP 算子衔接）。
     7. Transformer 里常见增强：SmoothQuant 把激活 outlier “搬到权重”，使 A 更容易 INT8；否则 attention 前后的 layernorm/残差会让 activation range 很难控。
  
  - W4A16（权重量化到 4bit，激活保持 FP16/BF16）流程：
     1. 选权重量化粒度：常用 per-channel + group-wise（如 group size 32/64），每组保存 scale（可选 zero-point）。
     2. 离线量化/打包：w_int4 按 nibble（4bit）打包存储，减少显存与带宽。
     3. 在线计算：激活 A 是 FP16/BF16；kernel 读取 packed int4 权重 + scale/zp，在寄存器/共享内存里解包并（显式或隐式）反量化成 FP16/BF16 fragment，再走 Tensor Cores 做 GEMM；或在支持的架构上用 int4 Tensor Core 路径并 FP16 accumulate（具体取决于硬件与 kernel 实现）。
     4. 误差控制：常见做 GPTQ（逐层/块优化权重，后量化误差更小）、AWQ（activation-aware，优先保留对激活敏感的权重通道/缩放策略），一般无需对激活做量化标定。
  
  - 应用场景与取舍：
     1. INT8（W8A8）：更适合 CNN/MLP/encoder 类模型或形状稳定、可充分融合的推理图；在 LLM 上通常需要 SmoothQuant/QAT 才能把精度做稳。收益来自更低带宽 + 更高 INT8 Tensor Core 吞吐，但代价是标定/精度风险、以及某些算子仍需 FP（layernorm/softmax）。
     2. W4A16：LLM 推理最常用的落地形态（尤其 decode memory-bound 场景），因为权重占用与带宽是大头，而激活保持 FP16 省掉 activation quant 的复杂度；对线上服务友好（少标定、精度更稳）。代价是解包/反量化开销、以及对 kernel/硬件支持依赖更强。

- Q：什么是 Two-batch overlap，什么场景 Two-batch overlap 是负优化？
   A：Two-batch overlap 是一种“双缓冲/双 batch 并行”手段：让 batch i 在 GPU 上执行时，同时在另一条 stream（或另一套 buffer/graph）准备并尽可能提前执行 batch i+1 的一部分工作，把原本串行的空洞隐藏掉。
  
  - 典型覆盖的“可 overlap”阶段：
     1. 调度/打包：continuous batching 的组 batch、padding/packing、KV block 分配与元数据准备；
     2. 数据搬运：H2D/D2H、小量 memcpy、KV swap/paging（若有）；
     3. GPU 前后处理：embedding 查表、rope/pos 处理、采样（部分实现可放 GPU）、以及下一步 decode 的轻量 kernel 链；
     4. CUDA Graph：用两套 graph/两套 buffer 轮换提交，降低 launch overhead，同时用 event 做依赖，达到“上一 batch 计算 + 下一 batch 准备”并行。
  
  - Two-batch overlap 可能负优化的场景：
     1. 资源已满：当前 batch 已经把 SM/DRAM/L2 吃满（compute-bound 或 memory-bound 都可能），再叠加下一 batch 只会产生资源争用，单个 batch 反而变慢；
     2. 同质瓶颈叠加：两条链路都 memory-bound（尤其 KV 读写+attention）时 overlap 会加剧带宽/缓存抖动，出现“看似并行、实际互相拖慢”；
     3. 形状被迫变小：为了做双缓冲把 batch 拆碎/强制更小的 GEMM/更碎的 kernel，算子效率下降（算术强度/融合机会变差）；
     4. 同步与管理开销：额外的 stream/event 同步、双份 buffer/graph 管理、KV 分配与回收复杂度上升，吞吐没提升但延迟/P99 变差；
     5. QoS 冲突：prefill 与 decode 混在同卡 overlap 时，prefill 大算子抢资源导致 decode tail latency 上升（线上更容易被判定为负优化）。

## 量化专题
- Q：PTQ 和 QAT 的区别？什么时候必须做 QAT？
   A：PTQ 是“训后量化”，不改训练流程；QAT 是“量化感知训练”，训练时插入 fake-quant/STE 让模型适应量化误差。经验上：W4A16 权重量化很多场景 PTQ 足够；W8A8（尤其注意力链路/激活量化）更容易需要 QAT 才能稳住精度。
  
  - PTQ：实现快、成本低；依赖 calibration 数据与量化策略（clipping/分位数/逐通道）选得好不好
  - QAT：成本高（要训练/微调），但能显著改善 activation outlier、分布漂移、以及量化噪声累积导致的退化
  
- Q：对称/非对称量化、per-tensor/per-channel/per-group 的取舍？
   A：口径可以按“是否需要 zero-point”和“scale 的粒度”来答。
  
  - 对称（symmetric）：通常 zp=0，实现简单、GEMM 更友好；权重量化常用
  - 非对称（asymmetric）：多一个 zero-point，更能贴合非零均值分布（激活更常见）；代价是算子实现更复杂，尤其是高吞吐 int8/int4 GEMM 的 epilogue
  - per-tensor：一个 scale（+zp），最快但误差最大
  - per-channel（权重常按输出通道）：误差显著更小，是 W8/W4 权重量化的常用选择
  - per-group（group-wise，如 32/64）：在 per-channel 的精度与 per-tensor 的元数据/访存开销之间折中，是 W4A16 的主流实现形态
  
- Q：量化里 scale/zero-point 怎么算？怎么防止溢出？
   A：典型做法：先确定浮点范围 [xmin,xmax]（或用分位数做 clipping），再把范围映射到整数可表示区间。
  
  - scale：s = (xmax - xmin) / (qmax - qmin)（非对称）或 s = max(|xmin|,|xmax|) / qmax（对称）
  - zero-point：zp = round(qmin - xmin / s)（非对称）
  - 溢出控制：clamp 到 [qmin,qmax]；对激活用 percentile clipping / smooth scaling 抑制 outlier
  
- Q：为什么 LLM 的 activation quant 更难？SmoothQuant/AWQ/GPTQ 各解决什么？
   A：LLM 的激活有更强的长尾/outlier（尤其 attention 前后、残差叠加后），直接 W8A8 会出现“少数 outlier 把 scale 拉大，导致大部分值分辨率变粗”的问题。
  
  - SmoothQuant：把激活的尺度通过逐通道缩放“搬到权重里”，让激活更容易 INT8，同时权重再按相应尺度反向调整
  - AWQ：找“对激活敏感”的权重通道/块，采用更合适的缩放或保留策略，核心是减少量化后损失的有效信息
  - GPTQ：逐层/逐块用二阶近似（Hessian/近似）优化量化误差的补偿，通常用于权重量化（W4/W8）精度更稳
  
- Q：推理端哪些算子通常不量化、仍保留 FP16/BF16/FP32？
   A：优先保证数值稳定与实现成本：layernorm/RMSNorm、softmax、部分 reduction/variance、以及一些激活函数的关键路径常保留 FP（至少在累加上用 FP32）。
  
  - 常见组合：线性层/MLP/投影做低精度（INT8/FP8/W4A16），归一化与 softmax 保持 FP16/FP32 accumulate
  - 评价口径：看误差累积最敏感的位置（attention 分布、norm 统计）先别动，先量化“吞吐贡献最大且稳定”的 GEMM
  
- Q：W4A16 的 int4 解包/反量化开销会不会抵消收益？怎么判断？
   A：decode 阶段往往 memory-bound，权重带宽是大头，因此 W4A16 通常仍净收益为正；但需要看 kernel 是否把解包/反量化融合进主循环并利用向量化加载与寄存器复用。
  
  - 净收益判断：对比 bytes/token 是否显著下降，以及 kernel 是否出现额外的 load/store 与指令开销导致 SM 利用率下降
  - 常见优化：权重按 group 预打包、scale/zp 与权重同向量加载、dequant 融合到 mma fragment 准备阶段、尽量避免中间 FP 写回
  
- Q：KV cache 量化怎么做？为什么它对长上下文很关键？
   A：KV cache 在 decode 是高频读，长上下文下几乎决定带宽与显存占用；量化 KV（如 K/V 用 FP8/INT8）能显著降低带宽与容量压力，但要处理 attention 分布敏感性与误差放大。
  
  - 常见做法：对 K/V 做 per-head 或 per-group scale；必要时对不同层/不同 head 用不同策略
  - 风险点：量化误差会影响注意力权重（softmax 前的 logits），更容易出现长上下文退化；需要更保守的 scale/clipping 或混合精度（部分层保留更高精度）
- Q：FP8 与 INT8 怎么选？面试时的回答框架是什么？
   A：按“硬件支持 + 算子形态 + 精度风险 + 工程复杂度”四点选。
  
  - FP8：更贴近训练链路（有成熟的 FP8 mixed precision 训练/推理路径），在 Hopper 等硬件上吞吐很强；但需要处理尺度管理（scale、amax history）与数值策略
  - INT8：推理生态成熟（尤其 CNN/MLP），对带宽收益明确；但 LLM 的 activation quant 难度更高，常需要 SmoothQuant/QAT 才稳
  - 工程落地顺序：先做 weight-only（W4A16/W8A16）→ 再做 activation-aware（W8A8/FP8）→ 最后考虑 KV quant 与全链路低精度

## 稀疏专题（结构化 / 半结构化）
- Q：对模型做结构化/半结构化稀疏，需要什么样的硬件支持？
   A：关键不是“有 0 就能跳过”，而是硬件/指令集能高效消费一种规则稀疏格式：稀疏模式固定、可预测，且配套有压缩布局与元数据（metadata）让 Tensor Core 在 MMA 级别直接做稀疏乘加。
  
  - 结构化稀疏（structured，典型是 block sparsity）：
     1. 稀疏模式：以固定 block（如 16×16/32×32）为单位保留/置零；
     2. 算子实现：更依赖 kernel/库对 block 索引与数据布局的优化（把“有效块”打包成连续、批量搬运与计算），硬件层面不一定需要专用稀疏 MMA 指令也能获得收益；
     3. 硬件友好点：block 内仍是致密计算（便于 Tensor Core），block 间用索引跳转减少无效搬运。
  
  - 半结构化稀疏（semi-structured，典型 N:M，如 2:4）：
     1. 稀疏模式：每 M 个元素里必须保留固定数量 N 个非零（例如每 4 个里保留 2 个）；
     2. 核心硬件支持：支持“带 metadata 的稀疏 MMA / Sparse Tensor Cores”，能在矩阵乘加时用 metadata 选择参与计算的元素；
     3. 约束：只对特定 dtype/shape/对齐要求有效（例如维度要是某些 tile 的倍数），很多框架会在不满足约束时回退到 dense；
     4. 软件栈：需要库提供压缩格式与高性能 kernel（常见是 cuSPARSELt / cuBLASLt 的对应路径或等价实现），不然稀疏带来的“算力省掉了”会被“索引/重排/访存碎片”抵消。
  
- Q：为什么半结构化稀疏经常要保存两套权重，并且会反复加载？
   A：因为“硬件高吞吐稀疏 GEMM”要求权重是压缩+metadata 的专用布局，而“训练/更新/某些回退路径”又需要可逐元素更新的 dense 权重；两种表示在存储布局、可更新性、以及能走的 kernel 路径上都不一样，所以工程上常同时维护。
  
  - 两套权重通常是什么：
     1. Dense master weights：用于 optimizer 更新（AdamW/momentum 等）、做 mask 更新/重剪枝、以及作为不满足稀疏约束时的回退路径；
     2. Sparse packed weights（+metadata）：用于半结构化稀疏 Tensor Core/GEMM 的高吞吐计算路径。
  
  - 为什么会“反复加载/转换”：
     1. 更新不可直接在 packed 上做：packed 权重通常是压缩后的非零值序列 + metadata，逐元素更新代价高且不适配常见优化器状态布局；
     2. 每次权重更新后都可能破坏 N:M 约束：为了继续走稀疏 Tensor Core 路径，需要重新选择每组保留的 N 个元素并重新 pack 生成 metadata；
     3. 形状/算子覆盖不全导致混合执行：有的层/shape 走 sparse，有的走 dense，运行时会在两套权重间切换，带来额外的加载与 cache 压力；
     4. 性能取舍：若不保留 packed，而是每次 matmul 前临时从 dense pack，会把大量时间花在重排/生成 metadata 上；因此常选择“持久化 packed”，但代价就是存储两份并在更新后同步。
- Q：Wan2.2 的权重稀疏化为什么能带来“实际加速”？加速收益来自哪里？
   A：能不能加速取决于你稀疏的“算子覆盖率 + 硬件路径 + 瓶颈形态”。在 Wan2.2 这类扩散/流匹配生成模型里，绝大多数时间花在 denoiser 主干（Transformer / MoE Transformer）的线性层/注意力投影等大 GEMM 上；如果这些 GEMM 能走半结构化稀疏（如 2:4）的稀疏 Tensor Core 路径，且权重以压缩格式常驻/复用，就能把 FLOPs 与权重带宽同时压下去，从而带来端到端收益。
  
  - 1）为什么稀疏能带来实际加速（必要条件）：
     1. 稀疏模式可被硬件直接消费：半结构化（N:M）更容易走“带 metadata 的稀疏 MMA”路径；不规则稀疏往往被索引/重排开销抵消；
     2. 高覆盖率：需要把稀疏施加在主耗时算子上（通常是 MLP 的两次大 GEMM、QKV/Out 投影、以及部分 cross-attn 投影），否则 Amdahl 定律会把收益吃掉；
     3. 形状足够大且满足对齐：小 GEMM/小 batch/小 token 会被 launch、epilogue、metadata 处理开销主导；
     4. 权重加载是显著成本：大模型每步会反复读取大量权重（扩散多步），如果权重压缩能减少 HBM→SM 的字节数，即便算子偏 memory-bound 也能见到收益。
  
  - 2）对 Wan2.2 来说“计算瓶颈占比如何”（面试口径）：
     1. 端到端通常由 denoiser 主干主导：文本编码（UMT5）与 VAE 编解码一般只占小头，迭代步数越多，主干占比越高；
     2. 主干内部：大部分时间落在 GEMM/attention（线性层与注意力投影）与少量归一化/激活/重排；因此权重稀疏化优先盯线性层的 matmul 路径；
     3. 判断方法：用 profiler 把时间按模块切开（text encoder / VAE / denoiser / sampler），再把 denoiser 内按算子（GEMM、softmax、layernorm、reshape）做 TopN，占比决定稀疏化的上限。
     参考：Wan2.2 使用 UMT5 文本编码、3D Conv VAE、以及用于生成的 Transformer/MoE Transformer 主干（A14B 系列每步约 14B 激活参数）。（deeplearning.ai 与 ModelScope 模型页）[https://www.deeplearning.ai/the-batch/alibabas-wan-2-2-video-models-adopt-a-new-architecture-to-sort-noisy-from-less-noisy-inputs/](https://www.deeplearning.ai/the-batch/alibabas-wan-2-2-video-models-adopt-a-new-architecture-to-sort-noisy-from-less-noisy-inputs/) ，[https://modelscope.ai/models/Wan-AI/Wan2.2-TI2V-5B-Diffusers](https://modelscope.ai/models/Wan-AI/Wan2.2-TI2V-5B-Diffusers)
  
  - 3）Wan2.2 的 T2V vs T2I（/TI2V）有什么区别（与稀疏收益相关的差异）：
     1. token 规模：T2V 多了时间维（多帧），latent token 数显著更大，attention/MLP 的总 GEMM 量随 tokens 近似线性增长；
     2. 瓶颈形态：T2I 更容易出现“算子更小 + launch/调度占比更高”，T2V 更容易成为“主干大算子占比更高”，因此稀疏化对 T2V 往往更容易体现端到端收益；
     3. VAE 开销：T2V 的 VAE decode/后处理也更重，但通常仍次于多步 denoiser 的总成本；如果你的流水里 VAE 占比被抬高，稀疏对端到端的贡献就会被稀释。
  
  - 4）加速收益究竟来自哪里（分解口径）：
     1. 省 FLOPs：对能走稀疏 MMA 的 GEMM，理论上乘加次数下降（例如 2:4 理论减半）；
     2. 省权重带宽：压缩后的权重 + metadata 使 HBM 读取字节数下降（扩散多步反复读权重时尤其关键）；
     3. 更高有效吞吐：稀疏 Tensor Core 在支持路径上提供更高的有效 TOPS，降低 compute 时间；
     4. 间接收益：权重更小带来 L2 命中改善、cache 压力降低、以及更少的显存占用（可能让 batch/并发更大，从而提升吞吐）。
- Q：对 Wan2.2 这类 DiT/Transformer denoiser 来说，attention 和 FFN 的计算量占比大概是多少？谁更容易成为瓶颈？
   A：没有固定比例，主要由 token 数 S（latent patch 数/帧数）与隐藏维度 d 决定。单层粗略 FLOPs：
  
  - Attention：约 4·S·d²（QKV+Out 投影） + 2·S²·d（QK^T + PV）
  - FFN：GELU(4d) 约 8·S·d²；SwiGLU 常见约 12·S·d²
  
  - 经验结论：
     1. S 较小（例如 T2I 的 token 规模，且 d 较大时）：通常 FFN 的 8·S·d² 占比更高，整体更 compute-heavy，瓶颈更常落在 MLP/GEMM；
     2. S 很大（例如多帧 T2V，token 数上万）：2·S²·d 会快速压过 FFN，attention 变成主要 FLOPs 与带宽瓶颈；
     3. 阈值口径：当 2·S²·d ≳ 8·S·d²（GELU FFN）时，约等价于 S ≳ 4d；若是 SwiGLU（12·S·d²），阈值更偏 S ≳ 6d。
  
  - 谁更“卡”取决于实现：
     1. attention 用 FlashAttention 后，memory/中间张量开销显著下降，但大 S 下仍可能被 S² 项主导；
     2. FFN 基本是大 GEMM，通常更接近 Tensor Core 峰值，稀疏/低精度对它的收益也更稳定。
- Q：FlashAttention 有没有办法“降低计算量”（FLOPs）？
   A：FlashAttention 本身主要是“降显存访问/降中间张量”，把 attention 从 memory-bound 拉回更接近 compute-bound；对标准 dense attention 来说，它并不改变主要 FLOPs 量级（仍需做 QK^T、softmax、PV），因此严格意义上不算“减少计算量”，而是“同样计算做得更快”。
  
  - FlashAttention 能带来的“计算侧收益”主要是有效吞吐提升：
     1. 不物化大矩阵（attention scores / P）：减少 HBM 读写与带宽瓶颈；
     2. block-wise online softmax：在块内完成 max/sum 归一化，避免全量存储；
     3. 更好的融合与流水：把 QK^T、softmax、PV 更紧密地融合，减少 kernel launch 与同步。
  
  - 真正“降低计算量”的路子（改变公式/稀疏化/近似）：
     1. 稀疏注意力：local/window、block sparse、top-k、分层稀疏（如 NSA/MoBA/DSA 一类），把 O(S^2) 降到 O(S·w) 或更低；
     2. 低秩/核方法近似：把 softmax attention 近似为可分解形式，降低 QK^T 的成本（但精度与稳定性要验证）；
     3. 降低 token 数：更强的下采样/patchify、latent 压缩、或对长上下文做 chunk/摘要（本质是减少 S）；
     4. 减少 heads/head_dim 或用 MQA/GQA：降低 QK^T 与 PV 的常数项（通常不改变 O(S^2) 但能显著降实际算量）。
  
  - 容易混淆的点：
     1. 低精度（FP16/FP8/INT8）通常不减少 FLOPs，但提高吞吐/降低带宽；
     2. KV cache 在 decode 阶段减少的是“重复算 K/V”，attention 仍需对历史做加权汇聚（单步仍是 O(S)）。
